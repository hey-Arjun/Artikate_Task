# Question B — Evaluating LLM Output Quality

## You have deployed a summarisation model for internal reports. Your manager asks: 'Is it performing well?' You need to answer this question rigorously, not with gut feel.


**Automated metrics**: I would compute standard reference-based scores such as ROUGE (n-gram overlap) and BLEU/METEOR as baselines, and semantic metrics like BERTScore. I can also use some specialized summarization metrics (e.g. MoverScore, SummaQA (QA-based), Sentence Mover’s); in fact, the SummEval toolkit includes 13 metrics (ROUGE variants, embedding-based scores, SummaQA, learned S3, METEOR, BLEU, CHRF, CIDER, etc.). However, these automatic scores have some limitations: they rely on reference summaries and capture only partial quality. For example, ROUGE/BLEU focus on n-gram overlap and often miss synonyms or paraphrases, yielding poor correlation with human evaluation. BERTScore and related embedding-based metrics handle better, but still don’t measure factual consistency. No single metric captures “goodness”: they can be supplemental checks. (I can also experiment with LLM-based scorers, e.g. prompting GPT-4 to rate summary relevance, coherence, factuality using rubrics, but must watch bias.)

**Human-evaluated ground truth** : I can build a test suite of internal reports with expert-written reference summaries. For example, sample ~100–200 recent reports and have domain experts or trained annotators produce concise summaries with strict guidelines (cover key points, avoid hallucination, preserve tone, etc.). Ideally each doc gets multiple references to account for variability. so we can pilot and refine the annotation rubric (clarify length, style, required content) to ensure consistency.

**Detecting regression**: first treat evaluation like automated testing. In our CI pipeline, whenever a model version changes (fine-tuning, prompt tweak, library update), we run the fixed test set through the new model and recompute metrics. We can use an automated framework (e.g. OpenAI’s Evals or custom scripts) that flags significant drops. For instance, track average ROUGE/BERTScore or percentage of human-evaluated summaries passing a quality threshold, and if they fall beyond a preset margin (e.g. >5% drop), alert developers. In practice we can also monitor trends (e.g. time-series plots of metrics after each release) and set statistical confidence bounds, so we can catch regressions early. at last also include targeted test cases (e.g. known tricky reports or edge cases) to guard against specific failure modes.

**Communicating quality**: To non-technical stakeholders,we can translate metrics into clear outcomes. For example, instead of “average ROUGE is 0.45”, we’d report “85% of key points in reports are captured on average,” or “90% of summaries were rated ‘coherent and accurate’ in blind human review.” Visual dashboards can show trends (e.g. a bar chart of average human-judged correctness or a line graph of metric scores over time) to show model stability. also complement scores with representative ex- show a few report-summary pairs marked “good” and “bad” to highlight successes and errors. Importantly,we can report error types (ex- “X% of summaries missed a critical fact” or “Y% had hallucinations”), so stakeholders see concrete issues rather than abstract metrics. If possible, collect user feedback or surveys (“satisfied/not satisfied”), creating a simple satisfaction KPI. In summary, the technical metrics feed a narrative for managers: “Our model now covers 40% more of the report content (via ROUGE/BERTScore trends), and human reviews find fewer coherence issues – we’ve reduced missing-critical-info cases from 15% to 5%.”This makes the concept easy to understand, even for someone without a technical background.
