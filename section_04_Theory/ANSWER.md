# Question B — Evaluating LLM Output Quality

## You have deployed a summarisation model for internal reports. Your manager asks: 'Is it performing well?' You need to answer this question rigorously, not with gut feel.


**Automated metrics**: I would compute standard reference-based scores such as ROUGE (n-gram overlap) and BLEU/METEOR as baselines, and semantic metrics like BERTScore. I can also use some specialized summarization metrics (ex- MoverScore, SummaQA (QA-based), Sentence Mover’s); in fact, the SummEval toolkit includes 13 metrics (ROUGE variants, embedding-based scores, SummaQA, learned S3, METEOR, BLEU, CHRF, CIDER, etc.). However, these automatic scores have some limitations: they rely on reference summaries and capture only partial quality. For example, ROUGE/BLEU focus on n-gram overlap and often miss synonyms or paraphrases, yielding poor correlation with human evaluation. BERTScore and related embedding-based metrics handle better, but still don’t measure factual consistency. No single metric captures “goodness”: they can be supplemental checks. (I can also experiment with LLM-based scorers, ex- prompting GPT-4 to rate summary relevance, coherence, factuality using rubrics, but must watch bias.)

**Human-evaluated ground truth** : I can build a test suite of internal reports with expert-written reference summaries. For example, sample ~100–200 recent reports and have domain experts or trained annotators produce concise summaries with strict guidelines (cover key points, avoid hallucination, preserve tone, etc.). Ideally each doc gets multiple references to account for variability. so we can pilot and refine the annotation rubric (clarify length, style, required content) to ensure consistency.

**Detecting regression**: first treat evaluation like automated testing. In our CI pipeline, whenever a model version changes (fine-tuning, prompt tweak, library update), we run the fixed test set through the new model and recompute metrics. We can use an automated framework (ex- OpenAI’s Evals or custom scripts) that flags significant drops. For instance, track average ROUGE/BERTScore or percentage of human-evaluated summaries passing a quality threshold, and if they fall beyond a preset margin (e.g. >5% drop), alert developers. In practice we can also monitor trends (ex- time-series plots of metrics after each release) and set statistical confidence bounds, so we can catch regressions early. at last also include targeted test cases (ex- known tricky reports or edge cases) to guard against specific failure modes.

**Communicating quality**: To non-technical stakeholders,we can translate metrics into clear outcomes. For example, instead of “average ROUGE is 0.45”, we’d report “85% of key points in reports are captured on average,” or “90% of summaries were rated ‘coherent and accurate’ in blind human review.” Visual dashboards can show trends (ex- a bar chart of average human-judged correctness or a line graph of metric scores over time) to show model stability. also complement scores with representative ex- show a few report-summary pairs marked “good” and “bad” to highlight successes and errors. Importantly,we can report error types (ex- “X% of summaries missed a critical fact” or “Y% had hallucinations”), so stakeholders see concrete issues rather than abstract metrics. If possible, collect user feedback or surveys (“satisfied/not satisfied”), creating a simple satisfaction KPI. In summary, the technical metrics feed a narrative for managers: “Our model now covers 40% more of the report content (via ROUGE/BERTScore trends), and human reviews find fewer coherence issues – we’ve reduced missing-critical-info cases from 15% to 5%.”This makes the concept easy to understand, even for someone without a technical background.


# Question C — On-Premise LLM Deployment

## A defence or government client requires that all AI inference run fully offline on their hardware — no API calls to external services. They have a single server with 2x NVIDIA A100 80GB GPUs and require responses within 3 seconds for a 500-token input.

**Model candidates**: With 2×A100 (80GB each) offline, we’d evaluate high-quality open models like Llama-2 (13B and 70B), Mistral variants (7B or 30B), or new open models (e.g. Gemma/Mixtral). Smaller models (7–13B) are easier to fit but may lag in complex reasoning; a 70B model yields better quality but is large. We note sizes: e.g. 13B model has ~13×10^9 parameters (~26–30 GB FP16), whereas a 70B model is ~140 GB FP16. So a 70B model won’t fit on one A100 in FP16 – we need to do either model parallelism or quantization (below). A 30B model (~60GB FP16) would fit on one GPU, leaving room for KV cache and batch; 7B (~14GB) is trivial.

**Quantization**: To meet latency and fit, we’d use 8-bit or 4-bit quantization. For example, int8 quantization (via GPTQ or bitsandbytes) roughly half memory, so 70B→70GB, which could span two GPUs. 4-bit (FP4/INT4) quantization (e.g. AWQ/GPTQ-4) further cuts memory, comes with minor accuracy loss for inference. We can also use NVIDIA’s FP8 (if TensorRT) which is half the size of FP16. **The trade-off**: 4-bit yields faster inference and lower memory, but we must need to validate that summary quality remains acceptable. In practice, an int8 or FP8 Llama2-13B ( ~13GB ) would fit comfortably on a single A100; Llama2-70B would need sharding or 2-GPU placement even in int8. We’d likely benchmark both a quantized 70B and a 13B model.

**Serving stack**: We’d deploy on GPUs (llama.cpp is CPU-only and not needed here). Two main options:

**TensorRT-LLM**: NVIDIA’s runtime for LLMs supports multi-GPU, FP8/INT4, optimized attention kernels and in-flight batching. It can split large models across GPUs (Tensor- or model-parallelism) and delivers maximum throughput. For ex- NVIDIA reports ~11,819 tokens/s on Llama-13B (single H200), implying ~1300–1500 tok/s on A100. Even a 70B (with two A100s) can reach a few hundred tokens/s. We can use TensorRT-LLM to meet the 3s/500-token requirement reliably.
**vLLM/HuggingFace Accelerate**: vLLM is designed for high-throughput GPU serving and supports quantization (INT4, INT8, FP8) and optimizes KV cache handling. With vLLM we can load HuggingFace models (including Llama/Mistral) and benefit from continuous batching. Throughput on A100 for, say, Llama-13B in FP16 or INT8 would be on the order of hundreds to thousands of tokens/s. vLLM also allows scaling across 2 GPUs easily. Another option is HuggingFace’s Optimum with Torch-TRT, but vLLM or TensorRT-LLM would outperform a plain Transformers pipeline.
We’d likely benchmark both. *In summary, with quantization and two A100s, throughput is ample: e.g. a quantized 13B model on one A100 can handle >1000 tokens/s. Even a 70B model would do ~300–400 tokens/s per GPU, so two GPUs could decode ~750–800 tokens/s, making a 500-token prompt plus output (~1000 tokens total) doable well under 3s. so we can also ensure batch size = 1 latency is low (TensorRT-LLM can hit a few hundred ms first-token time).*

**VRAM estimates**: For concreteness: Llama2-13B FP16 weights ~26 GB; adding KV cache (500 tokens, ~0.5–1 GB) stays under 30 GB, fitting on one A100. Llama2-70B FP16 ~140 GB, so we’d shard that across both A100s (70 GB each) or use int8 (~70 GB total) or int4 (~35 GB). Mistral-7B FP16 ~14 GB, trivial. We’d choose quantization levels to leave headroom for key/value caches and model overhead, aiming to use <80 GB/GPU. With these setups and frameworks (TensorRT-LLM or vLLM), we expect to meet the 3s/500-token SLA with comfortable margin.

**Limitations**: On-prem restricts us to available open weights. We must validate that quantization doesn’t drop accuracy unacceptably (first test on our evaluation set). Large models risk more latency variability, but multi-GPU and batching mitigate that. Also, extremely long contexts (>1000 tokens) could strain memory/cache; but our spec is 500-input tokens. Overall, this combination of 13B–70B models, 8/4-bit quantization, and GPU-optimized serving (TensorRT-LLM or vLLM) should achieve the latency target.
